# Bug 修复记录 — 2026-05-13

## 概述

本次修复了 SSA IR 流水线的 3 个关键 bug，使 `--ir` 回归测试从 18/20 通过提升到 20/20 全部通过。修复涵盖指针算术崩溃、递归函数返回错误结果两类问题。

---

## Bug 1：Load 代码生成覆盖活跃指针寄存器（影响指针解引用）

**症状**：`tmp_edge_pointer_arithmetic.c` 运行时崩溃，退出码 -1073741819（0xC0000005，Access Violation）。

**根因**：`IRCodeGenerator` 的 Load case 对 4 字节和 1 字节加载硬编码使用 `eax` 作为中间寄存器：
```cpp
if (ptrElemSize == 4) {
    loadReg = "eax";
    memPrefix = "dword ";
}
// ...
emitLine("    mov " + loadReg + ", " + memPrefix + "[rcx]");
```

当被加载的指针值在 `rax` 中时（如 GEP 结果），`mov eax, dword [rcx]` 会将 `rax` 的高 32 位置零，**覆盖仍被后续 Store 指令使用的指针值**。

具体崩溃路径：
```asm
mov rcx, rax           ; rcx = ptradd_13 (arr+12 的地址，在 rax 中)
mov eax, dword [rcx]   ; eax = 40，但 rax 被覆盖为 40！
movsxd rdx, eax        ; rdx = 40 (正确，deref_14 的结果)
; ...
mov rcx, rax           ; rcx = 40 (rax 已不是指针！)
mov dword [rcx], 42    ; 写入地址 40 → Access Violation!
```

**修复**：`src/backend/ir/IRCodeGenerator.cpp` — 直接使用 `movsxd`/`movzx` 内存操作数形式，消除中间寄存器：
```cpp
if (ptrElemSize == 1) {
    emitLine("    movzx " + dstReg + ", byte " + memAddr);
} else if (ptrElemSize == 4) {
    emitLine("    movsxd " + dstReg + ", dword " + memAddr);
} else {
    emitLine("    mov " + dstReg + ", " + memAddr);
}
```

**影响测试**：`minic_ir_pointer_arithmetic`。

## Bug 2：Value Slots 与 Callee-Saved 寄存器区域重叠（影响所有使用 callee-saved 寄存器的函数）

**症状**：`tmp_edge_recursive.c` 的 `factorial(5)` 返回 24（= 4!）而非 120（= 5!）。

**根因**：`generateFunction` 中 value slots 从 `slotOffset=0` 开始分配，第一个 slot 为 `[rbp-8]`。但函数序言中 `push rbx` 也将值保存到 `[rbp-8]`，两者**地址重叠**。

栈帧布局（以 factorial 为例，callee-saved: rbx, r12, r13, r14）：
```
[rbp]      = saved rbp     ← push rbp
[rbp-8]    = saved rbx     ← push rbx     ← 与 value slot 8 重叠！
[rbp-16]   = saved r12     ← push r12     ← 与 value slot 16 重叠！
[rbp-24]   = saved r13     ← push r13     ← 与 value slot 24 重叠！
[rbp-32]   = saved r14     ← push r14     ← 与 value slot 32 重叠！
[rbp-40]   = sub rsp 区域开始
```

当 `mov dword [rbp-8], r12d`（存储参数 n=5）执行时，覆盖了 `[rbp-8]` 处保存的 rbx。递归调用的 callee 虽然正确 push/pop rbx，但 pop 恢复的是被 n 覆盖后的值，导致 rbx 在乘法时不是 n 而是被破坏的值。

**修复**：`src/backend/ir/IRCodeGenerator.cpp` — 在分配 value slots 之前先收集 callee-saved 寄存器，将 `slotOffset` 初始化为 `calleeRegs.size() * 8`，跳过 callee-saved 保存区域：
```cpp
// 收集使用的 callee-saved 寄存器（必须在 value slot 分配之前）
calleeRegs.assign(regAlloc.usedCalleeSavedRegs.begin(),
                  regAlloc.usedCalleeSavedRegs.end());
std::sort(calleeRegs.begin(), calleeRegs.end());

// 初始化 slotOffset 跳过 callee-saved 寄存器保存区域
slotOffset = static_cast<int>(calleeRegs.size()) * 8;
```

修复后的栈帧布局：
```
[rbp]      = saved rbp
[rbp-8]    = saved rbx     ← push 区域
[rbp-16]   = saved r12
[rbp-24]   = saved r13
[rbp-32]   = saved r14
[rbp-40]   = 第一个 value slot  ← 不再重叠！
[rbp-48]   = 第二个 value slot
...
```

**影响测试**：`minic_ir_recursive`。

## Bug 3：storeResult 写 8 字节到 4 字节 Value Slot（潜在栈损坏）

**症状**：无直接测试失败，但会导致相邻 value slots 被意外覆盖。

**根因**：`storeResult` 对所有值统一使用 `to64(reg)` 写 8 字节到 value slot：
```cpp
emitLine("    mov [rbp-" + slot + "], " + to64(reg));
```

但 32 位类型的 value slot 只分配了 4 字节（`sz = inst.type.byteSize()` 后 `if (sz < 8) sz = 8` 实际上已修复此问题）。当 `to64("eax")` = `"rax"` 时，8 字节写入会覆盖紧邻的下一个 slot 的高 4 字节。

**现状**：由于 Bug 2 的修复将 slotOffset 初始化为跳过 callee-saved 区域，且所有 slot 最小为 8 字节（`if (sz < 8) sz = 8`），此问题在当前代码中不再触发。但 `storeResult` 应考虑使用与值类型匹配的写入宽度，以提高健壮性。

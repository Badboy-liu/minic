# Bug 修复记录 — 2026-05-11

## 概述

本次修复了 14 个回归测试失败，涵盖汇编语法错误、链接器符号导出缺失、优化器误删全局变量赋值、嵌套结构体初始化遗漏四类问题。

---

## Bug 1：全局变量缺少 `global` 指令（影响 9 个测试）

**症状**：链接器找不到 `gv_counter`、`gv_bytes`、`gv_x`、`gv_p`、`gv_fn_ptr` 等符号。

**根因**：`CodeGenerator::generate()` 只为函数和静态局部变量发射 `global` 指令，未为全局变量发射。NASM 中无 `global` 指令的符号对链接器不可见。

**修复**：`src/backend/CodeGenerator.cpp` — 在 preamble 生成中为非 extern 全局变量添加 `global` 指令。

**影响测试**：`minic_bss_integrity`、`minic_global_ptr_to_global`、`minic_global_ptr_to_string`、`minic_function_ptr_global`、`minic_function_ptr_table`、`minic_reloc_probe_global_ptr`、`minic_reloc_probe_function_ptr`、`minic_reloc_probe_string_ptr`、`minic_text_addr64_relocation`。

---

## Bug 2：结构体全局初始化缺少数据指令（影响 4 个测试）

**症状**：NASM 报 `instruction expected, found '19'`。

**根因**：`emitGlobalStructMemberValue()` 生成 `gv_g_pair: 19, 23`，缺少 `dd`/`dq` 等 NASM 数据指令。正确格式为 `gv_g_pair: dd 19, 23`。

**修复**：`src/backend/CodeGenInit.cpp` — 重写 `emitGlobalStructInitializer()`，将结构体展开为扁平的 `(directive, value)` 列表，按指令分组输出。新增 `emitGlobalStructScalars()` 替代原 `emitGlobalStructMemberValue()`。

**影响测试**：`minic_struct_global_init`、`minic_struct_global_ptr_init`、`minic_struct_init_nested_success`、`minic_struct_nested_init`。

---

## Bug 3：死存储消除误删全局变量赋值（影响 1 个测试）

**症状**：`minic_struct_global_ptr_init` 返回 41 而非 42。`g_value = 1` 赋值在 `-O2` 下被删除。

**根因**：`eliminateDeadStoresInBlock()` 将全局变量 `g_value` 当作局部变量处理。赋值后在同一函数内无读取，被误判为死存储。

**修复**：`src/backend/Optimizer.cpp` — 在死存储消除中检查 `VariableExpr::isGlobal`，跳过全局变量赋值。

**影响测试**：`minic_struct_global_ptr_init`。

---

## Bug 4：嵌套结构体本地初始化遗漏内层值（影响 1 个测试）

**症状**：`minic_struct_nested_init_local` 返回 3 而非 42。`struct Wrapper w = {{19, 20}, 3}` 中 `{19, 20}` 未被初始化。

**根因**：`emitLocalStructInitializer()` 对每个成员调用 `emitExpr()`，但 `visitInitializerListExpr()` 只返回 `xor eax, eax`（0）。嵌套初始化列表未被递归处理。

**修复**：`src/backend/CodeGenInit.cpp` — 当成员类型为结构体且初始化器为 `InitializerListExpr` 时，递归调用 `emitLocalStructInitializer()`。

**影响测试**：`minic_struct_nested_init_local`。

## IR 代码质量优化（2026-05-11 晚间）

在 SSA IR 流水线完成后，发现生成的汇编存在明显冗余。实施了 4 项优化：

### 优化 1：死 alloca 清除

**问题**：SSA mem2reg 后 alloca 无用但未删除，浪费 vreg ID 和物理寄存器。

**修复**：
- `SSAConstructor::removeDeadStores()`：删除无用 alloca 指令本身
- `IRDeadCodeElim::hasSideEffects()`：无 uses 的 alloca 不算有副作用

### 优化 2：ICmp+CondBr 冗余消除

**问题**：`if (z > 25)` 生成两层 ICmp：
```
%cmp_1 = icmp sgt %z, 25       ; 实际比较
%tobool = icmp ne %cmp_1, 0    ; 冗余布尔转换
br %tobool, ...
```
汇编表现为 `cmp+setg+movzx` → `cmp+setne+movzx+cmp rax,0+jne`，12 条指令。

**修复**（三层优化）：
1. **预扫描**：在代码生成前扫描所有 CondBr，识别将被 `generateICmpBranch` 消费的原始 ICmp，加入 `skippedICmps` 集合
2. **ICmp 跳过**：ICmp case 检查 `skippedICmps`（原始 ICmp）和 `uses.size() == 1`（tobool 模式），跳过发射
3. **`generateICmpBranch()`**：检测 tobool 模式时，从原始 ICmp 的操作数重新发射 `cmp+jCC`，跳过 `set+movzx` 链

汇编从 12 条减少到 4 条：`mov rax, r10; cmp eax, 25; jg if_then_2; jmp if_end_3`。死 `cmp_1` 物化被完全消除。

### 优化 3：立即数折叠

**问题**：`add eax, 10` 被展开为 `mov rcx, 10; add eax, ecx`。

**修复**：`generateBinaryOp()` 检查右操作数是否为小常量（-128~127），直接使用立即数形式。

### 优化 4：二元运算寄存器冲突修复

**问题**：`generateBinaryOp` 先加载 lhs 到 rax，再加载 rhs 到 rcx。当寄存器分配器将 rhs 分配到 rax 时，加载 lhs 会覆盖 rhs 的值。

**修复**：
1. 新增 `getPhysRegName()` 辅助方法，统一查询 IRInstruction 和 IRArgument 的物理寄存器分配
2. `generateBinaryOp` 和除法路径检查 rhs 是否在 rax 中：
   - 若是：先 `mov rcx, rax` 保存 rhs，再加载 lhs 到 rax
   - 若否：先加载 lhs 到 rax，再加载 rhs 到 rcx（原路径，无额外开销）
3. 修复 `regIndex()` 与 PHYS_REG_NAMES 顺序不一致的 bug

### 优化 5：Call 指令感知的寄存器分配

**问题**：线性扫描分配器未考虑 call 指令破坏所有 caller-saved 寄存器。当值跨越 call 时（如递归函数中的参数），被分配到 r11 等 caller-saved 寄存器会被 call 覆盖，导致错误。

**修复**：
1. `LivenessAnalysis` 记录所有 Call 指令的位置（`callPositions`）
2. `LinearScanAllocator::allocate()` 新增 `callPositions` 参数
3. 分配时检查每个区间是否跨越 call，跨越的区间强制溢出到栈

**效果**：递归函数 `factorial(n)` 中 n 正确溢出到栈（`mov [rbp-16], rcx`），call 后从栈恢复，不再被覆盖。

### 优化 6：汇编窥孔优化

**问题**：`storeResult` 写到分配器指定的物理寄存器后，`loadValToReg` 立即读同一值到另一寄存器，产生 `mov r10, rax; mov rax, r10` 冗余对。

**修复**：`emitLine()` 中实现窥孔优化器，缓存上一条输出指令。当检测到 `mov A, B` 后紧跟 `mov B, A` 模式时，跳过第二条冗余指令。

### 优化 7：函数调用地址加载简化

**问题**：`lea rax, [rel f]; mov r11, rax` 多了一次冗余搬运。

**修复**：直接 `lea r11, [rel f]`，每处函数调用省 1 条指令。

### 优化 8：二元运算寄存器感知优化

**修复**：`generateBinaryOp` 和除法路径检测 rhs 是否已在 rcx 中，若是则直接加载 lhs 到 rax，省掉 `mov rcx, rax`。

### 优化 9：调试输出清理

**修复**：所有 `[IRCodeGen]` 和 `[Liveness]` 调试输出用 `#define IR_CODEGEN_DEBUG 0` 门控，正常编译时无噪声。

### 测试结果

- 221 通过，0 失败，9 跳过（无回归）
- 修复了递归函数中 call 覆盖 caller-saved 寄存器的正确性 bug

## Bug 5：IRInstruction 从未分配 ID（影响整个寄存器分配）

**症状**：`--ir` 路径生成的汇编所有值通过栈 round-trip，`loadValToReg` 全部走 fallback `xor reg, reg`。

**根因**：`IRInstruction::id` 默认为 `-1`，`IRFunction::allocValueId()` 方法已定义但从未被调用。所有指令 `id = -1`，liveness 分析中 `vreg < 0` 条件将其全部跳过，导致活跃区间为空。

**修复**：`src/backend/ir/IRPrinter.cpp` — 在 `appendInstruction` 中为非 void 类型指令自动分配 ID：
```cpp
if (parentFunc && inst.id < 0 && inst.type.kind != IRTypeKind::Void) {
    inst.id = parentFunc->allocValueId();
}
```

## Bug 6：Liveness 分析不追踪 IRArgument 操作数（影响整个寄存器分配）

**症状**：SSA 构建后 `add %0, %1` 的操作数是 `IRArgument` 对象，但 liveness 分析只追踪 `IRInstruction` 操作数。参数值对寄存器分配器不可见。

**根因**：`computeLiveness` 和 `buildIntervals` 中操作数检查只做 `dynamic_cast<IRInstruction*>`，未处理 `IRArgument`。`IRArgument` 无 `id` 字段（只有 `argIndex`），无法直接参与 vreg 编号。

**修复**：
1. `src/backend/ir/Liveness.h` — 添加 `argVregStart`、`argVregMap` 字段
2. `src/backend/ir/Liveness.cpp` — 在 `compute` 中构建 `IRArgument* → vreg id` 映射；在 `computeLiveness` 和 `buildIntervals` 中追踪 `IRArgument` 操作数；参数从位置 0 开始活跃
3. `src/backend/ir/IRCodeGenerator.cpp` — 为参数分配 vreg ID（从 `func.nextValueId` 之后开始）

## Bug 7：参数 vreg 设置顺序错误（影响所有带参数的函数）

**症状**：`add(10, 20)` 返回 0。参数未从 ABI 寄存器（rcx/rdx）保存到分配的物理寄存器。

**根因**：`argVregStart` 和 `argVregMap` 在 liveness 分析之后才设置，但 `liveness.setArgVregStart(argVregStart)` 在分析之前调用，使用的是上一个函数的旧值。第一个函数（如 `add`）的参数 vreg 从 0 开始（上一个函数的残留值），而非 `func.nextValueId`。

**修复**：`src/backend/ir/IRCodeGenerator.cpp` — 重组 `generateFunction` 代码顺序：
1. 构建 `vregMap`（指令 → vreg id）
2. 设置 `argVregStart` 和 `argVregMap`（参数 → vreg id）
3. 调用 liveness 分析和寄存器分配

同时清除参数预分配栈槽，改为由寄存器分配器统一管理。

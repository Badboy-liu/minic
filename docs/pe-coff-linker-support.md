# PE/COFF Linker Support

This document describes the current built-in Windows PE/COFF linker in `minic`.

It is intentionally small and focused. The goal is not broad compatibility with arbitrary COFF producers. The goal is a clear model that matches the compiler's own NASM-generated AMD64 object files.

Linux executable linking is now a separate WSL-hosted system-linker path and remains outside the scope of this document.

Document type:

- long-lived support document
- describes the current linker behavior and supported boundary

For the overall project map, see [docs/README.md](README.md).

## Scope

The linker currently supports:

- Windows AMD64 PE executable output
- one or more NASM-generated AMD64 COFF object files
- mixed `minic`-generated objects and small hand-written NASM AMD64 COFF helper objects on the documented subset
- compiler-generated functions, globals, and entry point symbols
- a small DLL-aware import map across `kernel32.dll` and `msvcrt.dll`

The linker currently does not try to support:

- static libraries
- arbitrary third-party COFF object producers
- resources, TLS, unwind metadata, or debug sections
- Linux or ELF executable linking
- WSL-hosted Linux system-linker behavior, which is documented at the project level rather than as part of `PeLinker`

## Expected Inputs

The linker expects AMD64 COFF objects generated by the compiler's NASM backend.

In practice that means:

- machine type must be AMD64
- code lives in `.text`
- initialized globals may live in `.data` or `.rdata`
- uninitialized globals may live in `.bss`
- relocations are currently expected in `.text` and in the minimal `.data` cases produced by global pointer initializers
- the compiler provides a `mainCRTStartup` entry symbol in the object that owns `main`

If the object format or symbol conventions differ from what `minic` generates, linking may fail even if the object is valid COFF.

## Section Model

The linker recognizes and merges these sections:

- `.text`
- `.data`
- `.rdata`
- `.bss`

It also synthesizes:

- `.idata`
- `.reloc`

### Supported behavior

- `.text` is merged into the final executable code section.
- `.data` is merged as initialized writable data.
- `.rdata` is merged as initialized read-only data.
- `.bss` is treated as uninitialized data and contributes virtual size without contributing raw file bytes.
- `.idata` is generated by the linker for the minimal import table.
- `.reloc` is generated when the final image contains rebasing-sensitive absolute virtual addresses.

### Notes on `.bss`

The linker preserves section-relative addends for NASM COFF `REL32` references into `.bss`.

That matters for cases like:

- `int counter;`
- `char bytes[4];`

Both globals may be referenced through the `.bss` section symbol with different inline addends. The linker now preserves those addends so the final executable keeps distinct addresses for adjacent globals.

### Unsupported/ignored sections

Sections outside the linker's recognized subset are not part of the supported model. If a relocation or symbol depends on one of those sections, linking can fail.

## Symbol Resolution Model

The linker resolves:

- externally visible function symbols emitted by the compiler
- externally visible global variable symbols emitted by the compiler
- the generated program entry symbol `mainCRTStartup`
- the imported runtime termination function `ExitProcess`

### Multi-object linking

When multiple `.c` inputs are compiled, each file becomes its own object file. During final linking:

- defined symbols from all input objects are collected
- external references are matched against those collected definitions
- duplicate externally visible definitions are rejected
- unresolved external references are rejected

This is enough to demonstrate real linker behavior for small multi-file examples.

## Relocation Model

The current linker supports:

- AMD64 COFF `REL32` relocations applied from `.text`
- AMD64 COFF `ADDR64` relocations applied from `.text` on the current supported helper-object subset
- AMD64 COFF `ADDR64` relocations applied from initialized data sections in the current compiler-generated subset
- PE32+ `DIR64` base relocation entries synthesized for supported absolute-address image slots

This is the main relocation shape used by the compiler's current code generation path.

### Supported targets

A supported `REL32` relocation may target:

- a symbol defined in a recognized merged section
- an external symbol resolved from another linked object
- the special imported `ExitProcess` thunk

A supported `ADDR64` relocation may currently target:

- a recognized merged-section symbol referenced directly from `.text`
- a compiler-generated global object address stored in `.data`
- a compiler-generated string literal address stored in `.data`
- a compiler-generated function address stored in `.data`

### Base relocation model

Whenever the linker writes a final absolute image virtual address into section contents, it now also records that slot for PE rebasing.

For the current supported subset, that means:

- compiler-generated `.data` pointer initializers
- supported `.text` `ADDR64` helper-object cases

Those slots are emitted into `.reloc` as `IMAGE_REL_BASED_DIR64` entries grouped by 4 KiB page RVA.

### Important detail: section symbol plus addend

NASM may encode references to section-relative data by:

- targeting a section symbol such as `.bss`
- storing the intra-section offset as the inline relocation addend

The linker must preserve both parts:

- the target section RVA
- the original signed addend already present at the relocation site

This behavior is now part of the supported model and is exposed by `--link-trace`.

### Current data-relocation boundary

The linker now supports the smallest data-side relocation subset needed by the current compiler:

- `int *p = &x;`
- `char *p = "A";`
- `int (*p)() = f;`
- `int (*table[2])() = { f, g };`

In NASM-generated AMD64 COFF objects, those cases currently appear as:

- `.data` relocation type `ADDR64`
- targeting a recognized merged section such as `.bss` or `.rdata`
- with the final symbol offset carried through the stored addend when needed

### Unsupported relocation behavior

The linker does not currently claim broad relocation coverage beyond this minimal `REL32` model. New relocation kinds should be treated as unsupported until explicitly added and documented.

## Import Model

Import support is intentionally minimal, but it is now table-driven inside the linker.

The linker synthesizes a PE `.idata` section that provides:

- import descriptor data
- import lookup table data
- import address table data
- one or more imported function names
- one or more DLL names
- import thunks in `.text` for imported call targets

Currently the built-in import catalog includes:

- `ExitProcess` from `kernel32.dll`
- `GetCurrentProcessId` from `kernel32.dll`
- `puts` from `msvcrt.dll`
- `putchar` from `msvcrt.dll`
- `printf` from `msvcrt.dll`

This is enough to demonstrate:

- the compiler-generated `mainCRTStartup` wrapper terminating through `ExitProcess`
- a minimal imported Win32 API call
- a minimal imported C runtime call

The linker also looks for an additive repository catalog at `config/import_catalog.txt`. When present, each non-comment row must use:

```text
symbol|dll|import
```

For example:

```text
fn_strlen|msvcrt.dll|strlen
```

The current merge rule is intentionally strict:

- built-in entries always remain available
- file-backed entries may add new symbols
- file-backed entries may not override built-in symbols
- duplicate symbols inside the file are rejected

This is still not source-level user syntax. It is a repository-side catalog extension point.

Internally, the mechanism is now split into:

- a built-in import catalog
- an optional file-backed import catalog
- an import resolver that matches unresolved extern symbols to catalog entries
- an `.idata` layout builder that only consumes resolved import items

This keeps the current source internal while leaving a clean seam for future user-defined import sources.

### `printf` boundary

`printf` support in this phase is intentionally conservative.

What is supported:

- linking `fn_printf` through `msvcrt.dll`
- simple repository-level calls such as `printf("value=%d\n", 42);`

What is not claimed:

- general C varargs completeness
- arbitrary `printf` usage beyond the cases actively covered by regression tests

## Executable Layout Model

At a high level, the final image is laid out as:

1. PE headers
2. merged `.text`
3. merged initialized data sections
4. merged `.bss` as virtual-only uninitialized storage
5. synthesized `.idata`

Two address concepts are important:

- `RVA`
  Relative virtual address inside the loaded image
- `raw file pointer`
  Offset inside the on-disk executable file

For initialized sections, both concepts matter.

For `.bss`, virtual size matters but raw file size is zero, because the loader provides zero-initialized memory at runtime.

For the current supported `ADDR64` subset, executables no longer rely on a fixed preferred image base. The linker now emits a PE `.reloc` table so loader-applied rebasing can adjust stored absolute addresses correctly.

## Link Trace

The linker now supports:

- `--link-trace`

Example:

```powershell
.\build\Debug\minic.exe .\input\tmp_bss_integrity.c --link-trace --keep-obj
```

The trace prints six high-level blocks:

1. input objects
2. merged sections
3. imports
4. resolved symbols
5. relocations
6. base relocations

The trace is meant to explain what the linker is doing, not to dump the entire binary.

Each block now begins with a short summary line so the trace is easier to skim before reading the detailed entries.

In the input-object block, the linker now also prints the externally visible symbols and unresolved extern references that matter to the current supported subset.

## Automated Regression

The project now provides a small automated regression entry through CTest.

Fastest commands:

```powershell
ctest --preset phase-current
ctest --preset bss
ctest --preset linker-failures
```

Run:

```powershell
ctest --test-dir .\build -C Debug -L phase-current --output-on-failure
```

The regression currently covers:

- baseline single-file code generation and linking
- `.bss` integrity with `--link-trace`
- multi-object PE linking with `--link-trace`
- mixed C plus NASM-object linking on the documented subset
- global pointer initializer relocations with `--link-trace`
- function-pointer and function-pointer-table relocations with `--link-trace`
- DLL-aware imports from `kernel32.dll` and `msvcrt.dll`
- explicit linker-failure cases such as duplicate symbols and unsupported relocation/section shapes

Each regression case is declared in `CMakeLists.txt`, where the test lists:

- source file inputs
- compiler arguments such as `--link-trace`
- output executable path
- expected exit code
- required trace or compiler output markers

Those parameters are passed into `tests/run_regression_case.ps1`, which performs the compile-and-run check for one case.

## Common Failure Modes

The current linker is expected to fail clearly for cases such as:

- missing `mainCRTStartup`
- unresolved external symbol
- duplicate external symbol
- unsupported COFF relocation type
- unsupported target section in a relocation
- object metadata that is truncated or too small to be valid COFF

These failures are part of the current documented boundary. The diagnostics now prefer a `linker diagnostic:` prefix and include the relevant object path, symbol, or section when that context is available.

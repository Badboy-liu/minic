# Global Pointer Initializer Relocations Design

Date: 2026-04-28

## Goal

Extend the current `minic -> NASM -> COFF -> PE` pipeline so initialized global data can store symbol addresses in the smallest useful subset needed by `minic`.

This phase should make the following kinds of programs work:

```c
int x;
int *p = &x;
```

```c
char *p = "A";
```

The goal is not to implement general-purpose data relocation support. The goal is to cross the next real compiler/linker boundary in a controlled, teaching-friendly way.

## Why This Phase Next

The current relocation matrix shows that the existing `minic` pipeline is still mostly a one-relocation-model system:

- relocations originate from `.text`
- the linker handles AMD64 COFF `REL32`
- all important non-local references are currently computed in code

The next meaningful step toward a more complete C compiler is to allow data to contain symbol addresses, not just code.

That is the smallest next step that:

- expands real C expressiveness
- keeps the project self-consistent
- forces the linker to grow in a meaningful but still controlled way

## Scope

This phase supports only the smallest practical subset of initialized global address constants.

### In Scope

- global pointer initialized with the address of a global object
- global pointer initialized with the address of a string literal
- linker handling for the minimal data-section relocation shapes produced by those inputs
- teaching trace and regression coverage for those cases

### Out Of Scope

- general global constant-expression evaluation
- function pointer initializers
- pointer arrays or richer address tables
- arbitrary address arithmetic in global initializers
- a broad data-relocation framework for all future cases
- generic COFF compatibility goals beyond `minic`'s own outputs

## Target Examples

### Example 1: Pointer To Global Object

```c
int x;
int *p = &x;

int main() {
    *p = 42;
    return x;
}
```

### Example 2: Pointer To String Literal

```c
char *p = "A";

int main() {
    return p[0];
}
```

These two examples should drive the whole design.

## Current Limitation

Today the compiler allows only these global initializer families:

- integer constants for integer globals
- string literals for `char[]`

It does not yet support a global pointer storing the address of:

- another global symbol
- a string literal symbol

Even if the front end accepted these forms, the linker is currently shaped around relocations originating from `.text`, not `.data`.

## Proposed Behavior

### Front-End Behavior

The semantic rules for global initializers should expand in one narrow way:

- if the declared global type is a pointer type, accept an address constant initializer in the supported subset

Supported address constants in this phase:

- `&global_object`
- string literal expression

Rejected for now:

- `&function`
- pointer arithmetic such as `&x + 1`
- mixed arithmetic expressions
- casts
- non-address global initializers for pointer types unless already supported by current rules

### Code Generation Behavior

For supported pointer initializers, code generation should emit initialized storage in `.data` that refers to a symbol address rather than embedding a plain integer literal.

Conceptually:

- pointer-to-global should emit a data slot initialized with the target global symbol
- pointer-to-string should emit a data slot initialized with the generated string label in `.rdata`

The exact NASM syntax should be chosen based on what produces a stable and inspectable COFF relocation for the current target.

### Linker Behavior

The PE linker should grow from:

- reading relocations only from `.text`

to:

- reading the minimal relocation subset needed from initialized data sections produced by these pointer initializers

This phase should stay narrow:

- support only the real relocation forms generated by the compiler for the in-scope examples
- reject anything else clearly

## Design Constraints

### 1. Evidence Before Generalization

The linker change must be driven by real object output from NASM.

That means implementation should first inspect the object files generated by the two target examples and answer:

- which section holds the relocation
- which relocation type NASM emits
- whether the target is a section symbol or a named symbol
- whether an addend is already embedded in the section bytes

Only then should the linker be extended.

### 2. Keep The Teaching Model Intact

This change should not turn the linker into a broad, implicit relocation engine.

Instead, the teaching story should stay crisp:

- today `.text` uses `REL32` for non-local references
- this phase adds the first data-stored address references
- trace output should make that visible

### 3. Avoid Premature General Frameworks

Do not introduce a large relocation abstraction unless the real object evidence requires it.

Prefer:

- a small extension for the exact current need

over:

- a speculative relocation system designed for many future cases

## Compiler Pipeline Changes

### Semantic Analysis

The semantic analyzer should distinguish between:

- integer constant global initializers
- `char[]` string-literal initializers
- pointer address initializers in the supported subset

It should reject unsupported pointer initializer forms with a clear diagnostic.

### AST / Representation

The existing AST may already be enough if pointer initializers can still be represented as existing expression forms.

If the back end needs a clearer distinction between:

- numeric data initializer
- symbol-address data initializer

then the internal global representation should be extended in the smallest possible way.

### Code Generator

The global data emission path should gain a new branch for pointer globals with supported address constant initializers.

It should emit data records that cause NASM to place a real relocation into the object file rather than flattening the address to an immediate number.

### PE Linker

The linker should:

- detect the relevant initialized-data relocations
- resolve them against the already-supported symbol model
- patch the final merged `.data` bytes correctly

The implementation should remain explicit about which section and relocation forms are supported.

## Trace Requirements

`--link-trace` should remain concise, but this phase should add enough visibility to teach the new behavior.

Minimum expectation:

- input-object summary shows that the object contains the relevant data section
- resolved-symbol summary still shows the target symbols
- relocation summary should include the new data-side relocation entries once they are supported

The trace should make it obvious that the address now lives in data, not only in code.

## Regression Requirements

This phase is not complete without automated coverage.

Add at least:

- one regression case for pointer-to-global
- one regression case for pointer-to-string

Each should:

- compile through the full pipeline
- run the produced executable
- assert the expected exit code

Where useful, the regression should also assert trace markers that show the intended relocation path.

## Risks

### Risk: NASM Emits A Different Relocation Shape Than Expected

This is the highest-risk unknown.

Mitigation:

- inspect the real object output first
- design to the observed relocation form, not to assumptions

### Risk: Scope Expands Into General Constant Expressions

This phase can easily grow too large if pointer initialization rules are not kept narrow.

Mitigation:

- support only the two target initializer families
- reject everything else clearly

### Risk: Linker Abstraction Grows Too Early

A future-proof relocation framework may be tempting, but this phase should stay grounded in the smallest real requirement.

Mitigation:

- implement only what the target examples require
- document remaining unsupported cases explicitly

## Success Criteria

This phase is complete when:

- `int *p = &x;` style programs compile, link, and run correctly
- `char *p = "A";` style programs compile, link, and run correctly
- the linker handles the exact minimal data relocation forms needed for those programs
- `--link-trace` exposes the new relocation path clearly enough for teaching
- automated regression tests cover both new cases
- unsupported pointer global initializer forms still fail clearly

## Follow-On Work

After this phase, the most natural next expansions are:

- function pointer initializers
- pointer arrays / address tables
- broader `.rdata` and `.data` relocation coverage
- richer import-address usage in data
